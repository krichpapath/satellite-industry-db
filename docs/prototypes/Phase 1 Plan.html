§01

Inherited context

What Phase 0 left us with, in two paragraphs.

The Inception Report defines a central national database for Thailand's satellite industry whose job is to absorb scattered records — from BOI, DBD, NRIIS, GISTDA, research institutes and private firms — and re-publish them as one standardized, governed, queryable corpus. The system has three architectural tiers (Presentation / Application / Data) and rests on five named design principles: Data Integration, Data Transformation, Data Standardization, Data Governance, Scalability. The schema is anchored on Firm-Level Information; six additional domains attach via relational keys.

Phase 1 builds the Presentation tier only, with no real backend, database, API, or auth. It ships as a Vercel-deployable frontend prototype whose success metric is explicitly user-experience appeal for the data-planner persona — not feature completeness. The deliverable is the planner's workbench at high fidelity: forms, bulk import, validation feedback, source attribution, and seven-domain navigation, all populated by mock data. Public dashboards, search UX, value-chain network views and ecosystem analysis screens are deferred to Phase 2+.

Continuity from Phase 0 Eight planner flows, seven data domains, ten must-have features (M1–M15), and five engineering principles carry forward unchanged. The five open content questions (sub-segmentation taxonomy, juristic-ID confirmation, approval depth, confidentiality tiers, priority source agencies) are deferred by user direction — Phase 1 uses defensible defaults and ensures no UI commits to a wrong answer. See §13.
§02

Decisions fixed for Phase 1

What we are no longer debating, with the one-line tradeoff for each.
Stack
Next.js 14 (App Router) · TypeScript · Tailwind CSS. Tradeoff: more setup ceremony than Vite, but App Router gives us route groups, layouts, and serverless routes for free — exactly the seams Phase 2 needs.
Hosting
Vercel. Preview deployment per branch. Custom domain optional, not in Phase 1 scope.
State
URL state for filters/tab/firm-id · React Hook Form for forms · Zustand for cross-page draft & import-job state · localStorage for session persistence. Tradeoff: three stores not one, but each lives where the access pattern is, so swap-cost in Phase 2 is per-feature, not whole-app.
Forms & validation
React Hook Form + Zod. Zod schemas are shared between client and the future server (M6 in the Phase-0 must-haves). Tradeoff: Zod is the strongest seam we can build now — the rules are portable to a Phase-2 Node/Edge server unchanged.
Tables
TanStack Table v8 + headless rendering with custom Tailwind cells. Tradeoff: more cell-level code than AG Grid, but it stays inside our design system and ships ~30kb instead of ~300kb.
Bilingual
Full TH/EN parity. Two layers, intentionally separate: UI chrome i18n via next-intl (dictionary files, header toggle); field-level bilingual data via name_th/name_en pairs in the schema, rendered with a small <BiText> component that picks active or both.
Roles in demo
Single role — Planner. Approval queue / reviewer-diff (Flow 9) deferred to Phase 1.5.
Mock data
Generated with @faker-js/faker + a curated Thai-satellite-adjacent firm-name seed list. ~40 firms total, 6 fully populated across all 7 domains.
Density
Hybrid: compact tables (24–28px rows), comfortable forms (16px base, 12px field gap).
Variations
Three visual directions explored side-by-side in Visual Directions.html (separate deliverable). Final prototype commits to one — chosen by you.
Dark mode
Recommendation: light primary, dark optional unless visual direction B (Mission Console) wins — in which case dark becomes primary. Decision deferred until §06 is reviewed.
Open content questions
Deferred. Defaults documented in §13.
§03

Frontend architecture

Three layers, one mock seam, zero coupling to the future backend.

Layered shape

  1. Routes & pages — Next.js App Router. Each page is a thin shell that composes feature modules.
  2. Feature modules — one folder per data domain (features/firm, features/valuechain, …). Each owns its forms, Zod schemas, table columns, and feature hooks. This is the unit of ownership and the unit Phase 2 will refactor when the API arrives.
  3. Data-access layer — every screen consumes useFirm(id), useFirms(filter), useImportJob(), etc. In Phase 1 these read from in-memory + localStorage; in Phase 2 they swap to fetch against real endpoints. No screen ever reads the mock store directly.
  4. UI primitives — the design-system components (Button, Input, Select, DataTable, FieldGroup, Drawer, Toast, Stepper, EmptyState, Skeleton, StatCard, BiText, ProvenanceCard).

Three-tier mapping (Phase 1 vs Phase 2)

Presentation

Pages, feature modules, UI primitives. Built in Phase 1.

Application (Phase 1 stub)

Zod validators + mock service functions. Same function signatures the real Node service will expose. Built in Phase 1.

Data (Phase 1 fake)

In-memory store + localStorage. Phase 2 replaces with Postgres + Prisma; hook signatures unchanged.

Three-tier credibility cue Because the Inception Report places validation in the Application layer, the mock service functions deliberately simulate a 120–300ms latency and surface server-side error codes (DUPLICATE_DETECTED, UNKNOWN_SEGMENT_CODE, …). The UI renders skeletons during the wait and shows server codes alongside inline messages. This makes the architecture visible in the demo without building real servers.

§05

Mock data-access layer

The single seam between Phase 1 and Phase 2. Designed to swap, not to be rewritten.

Hook contract

Every page consumes hooks with shapes identical to what a SWR/React-Query layer over a real REST API would expose. Phase 2 replaces only the body of lib/data/service/*; hook signatures and return shapes do not change.

// lib/data/hooks.ts — examples
useFirm(id: string)                  → { data: Firm | undefined, status, error, mutate() }
useFirms(query: FirmsQuery)          → { data: Firm[], total, page, status, error }
useFirmDraft()                       → { draft, set(patch), submit(), reset() }
useFuzzyDupeMatch(input: NameOrJID)  → { matches: FuzzyHit[], status }
useImportJob(jobId?)                 → { job, parse(file), dryRun(), commit() }
useTaxonomy(name: 'trl'|'valuechain'|'isic'|'esg_cert'|…) → { options, status }
useSources()                         → { data: Source[], status }
useAuditLog(firmId)                  → { entries: AuditEntry[], status }

Simulated latency & failure modes

  • All mock service functions await sleep(rand(120, 380)) before returning — felt as the "server thinking" moment that Phase 0 placed in the Application layer.
  • One in every N writes (configurable; default 0, demo mode 1-in-12) returns a synthetic SERVER_VALIDATION_FAILED error so the demo can show the "rerun on the server" round-trip. Off by default — toggle via dev menu.
  • Duplicate-detection on juristic-ID is deterministic against seed data: typing a juristic-ID that prefixes one of the seeded firms produces a real, scored fuzzy hit.

Persistence

  • Seed JSON loads on first visit; subsequent CRUD writes a JSON diff overlay into localStorage under satdb:overlay.
  • "Reset demo data" button in the header dev menu wipes the overlay and rehydrates — useful for live demos.
  • Import jobs persist under satdb:import:<jobId> — closing the tab mid-import and reopening it lands back on the dry-run review screen.
§06

Design system — 3 directions to choose from

Full mocks of each direction live in Visual Directions.html. Below: the strategic summary.

A · Calm Authority

GOV / ACADEMIC · LIGHT

BOI / NSTDA annual-report energy. Parchment surfaces, deep navy, maroon accent for required/errors. Plex Serif headings + IBM Plex Sans + Sarabun. Reads as credible institution.

B · Mission Console

AEROSPACE / TECHNICAL · DARK

ESA/SpaceX dashboard energy. Slate surfaces, cyan primary, amber warnings, mono numerics. Space Grotesk + IBM Plex Mono + Noto Sans Thai. Reads as operational, technical, instrument-panel.

C · Editorial Data

PUBLICATION · LIGHT

FT / Bloomberg / OECD report energy. Cream surfaces, ink, deep teal accent, occasional rust. Source Serif + IBM Plex Sans + Sarabun. Reads as data publication, journalistic seriousness.

Common to all three directions

  • Typography stack always paired with a Thai face. Sarabun for A and C (matches Plex's x-height); Noto Sans Thai for B (matches Space Grotesk's geometric feel). Thai-only headings use the Thai face at its native weight, never forced to match Latin metrics with bad letterspacing.
  • Density tokens. Compact (24px row) and Comfortable (32px row) variants. Tables default compact; forms default comfortable.
  • Icon set. Lucide — Apache 2.0, comprehensive, line-weight matches all three directions. No emoji.
  • Motion. 120ms ease-out for entrances, 180ms ease-in-out for state changes. Reduced-motion media query respected.
  • Accessibility floor. WCAG AA contrast in both themes; focus rings always visible; all interactives keyboard-reachable; aria-live on validation feedback.
Recommendation For an audience of BOI / NSTDA / GISTDA stakeholders, Direction A (Calm Authority) is the safest credibility play, and Direction B (Mission Console) the strongest "this is a domain product, not a CRUD admin" statement. Direction C is the bold third option — risk: editorial polish reads as "magazine" rather than "system" to government audiences. My vote, if pressed: A as base, with B's data-density tropes (mono IDs, status chips) selectively borrowed.

Hero storyboard — Flow 1: Onboard a new firm

The screen that makes the case that this is not a CRUD admin.

The planner has a new firm to add — name on a BOI announcement, or a referral from GISTDA. The hero arc is six frames. Validation moments are marked.

01Open
Workbench home → "Add new firm"

Workbench shows: 3 most-recent firms, 1 in-progress import job, 2 drafts. Single primary CTA "Add new firm" with shortcut hint N. Microcopy: "Start with the juristic ID — we'll check what's already in the system."

  • Keyboard: N opens the form. / opens command palette.
  • Empty state: if no data ever, copy says "ยังไม่มีบริษัทในระบบ — เริ่มต้นด้วยเลขทะเบียน 13 หลัก" / "Nothing here yet — start with a 13-digit juristic ID."
02Identify
Juristic-ID-first form (the "credibility moment")

Single full-width field with a 0123456789012 placeholder. Other fields are disabled until juristic ID validates regex ^\d{13}$. On 13th digit, an inline simulated DBD-lookup spinner runs ~280ms and either (a) returns existing firm → suggest "Link to existing" instead of create, or (b) returns clear → unlocks the rest of the form.

Validation moments regex on each keystroke (silent until 13 chars); DBD-lookup on completion; on success, the locked fields fade in with 80ms stagger.

  • Success state: green check next to the ID + small text "ไม่พบในระบบ — กรอกต่อได้" / "Not in system — proceed".
  • Match state: a FuzzyMatchList appears below with scored hits ("0.94 — Thaicom Public Co., Ltd. · last updated 12 days ago by faker-name"). "Use existing" / "Continue creating new (override)" actions.
  • Error state: 13-digit-but-invalid checksum → server error pill "INVALID_JURISTIC_ID_CHECKSUM" + plain-language line.
03Describe
Identity payload — bilingual name pair, address, dropdowns

Form is comfortable-density: 2-column grid. Bilingual pairs (name_th/name_en) sit side-by-side with a small "swap auto-fill" affordance for the common case where TH is provided and EN is romanized. Ownership type, industry type, founded year. Lat/lon optional with a "drop on map" placeholder (map deferred).

Validation moments EN-only ASCII / TH-only on respective fields; founded_year ∈ [1957, current_year]; email format on blur; required fields marked with maroon dot, not with red border, until submit.

04Attribute
Provenance — the always-on right rail

The right rail is non-collapsible during create. Three required selections: Source (typeahead against useSources() with "add new" inline), Confidence (reported / estimated / derived — radio), Reporting period (defaults to current year). This is the same component that will appear on every domain tab — built once.

Why it's persistent Phase 0 made source attribution mandatory on every record. Making the panel persistent is the visual proof that governance is first-class, not a checkbox at the bottom.

05Commit
Save → simulated server validation → seven-domain landing

Submit triggers ~280ms server-validation skeleton in the right rail ("Application layer validating…"). On success, a toast slides in ("Firm created · 1 source attached · 7 domains awaiting input"), the URL updates to /firms/[id]/overview, and the screen reveals the new seven-domain shell with the sidebar showing 7 completeness rings each at 0% except Identity at ~30%.

Microcopy direction precise, calm, declarative; never "Yay!" or emoji; always cites the underlying action.

06Tour
Seven-domain shell — the relational-model-made-visible

Single page, left sidebar lists the 7 domains with completeness rings. Active tab = Overview, showing identity summary, provenance card, and 7 stat cards (one per domain) reading "0 records" with a "+ Add" affordance. The seven-domain layout is the central visual metaphor of the product — every other page extends or modulates it.

Keyboard 1–7 jumps to each domain. G then F back to firms. E edit.

Empty & loading states

  • Loading: right-rail shows a skeleton card matching ProvenanceCard's exact shape. Form body fades opacity slightly. No spinners — only skeletons for content, micro-spinners for inline lookups.
  • Empty: the seven-domain shell on a brand-new firm shows muted "0 records — Add the first" cards in each domain stat; no apology copy, no illustrations.

§09

Supporting storyboards

Three more flows at storyboard depth. Remaining flows (HR, investment, ESG, audit) get representative-screen depth in implementation.

Flow 7 — Bulk import (throughput proof)

01Select
Domain template picker

Stepper across the top: Upload → Validate → Review → Commit. Step 1: pick which domain template (Firm / Value Chain / Technology / …). Each card shows column count, example row, and "Download template (.xlsx)".

02Drop
FileDrop with preview

Large dashed dropzone. On drop: file metadata (rows, encoding, header match), then a preview table of the first 20 rows. Encoding-mismatch error if not UTF-8 — explicit, because Phase 0 named this as a Thai-data risk.

03Dry-run
Validation report — the wow moment

Full-table view of every row with two added columns: _row_status (OK / WARN / REJECT) and _error_codes (chips like DUPLICATE_DETECTED, UNKNOWN_SEGMENT_CODE). Top of screen: counts (328 OK · 17 warn · 5 reject). Click any row → drawer with the failing field highlighted + fix-in-staging inline editor.

Microcopy "We caught 5 problems before they reach the database. Fix here or commit the OK rows only."

04Fix
Inline staging edit

Edit a cell in staging. Re-validates that row only. _row_status flips from REJECT to OK. Mock service uses the same Zod schema as the create form — the demo's promise that "validation rules are portable" becomes visible.

05Commit
Selective commit

Three commit options: Commit OK only (default), Commit OK + WARN, Commit all (force) — last requires typed confirmation. Progress bar with row counter; on done, link to created firms in the directory + downloadable "import_batch_xxx.csv" with statuses.

Flow 5 — Supply chain (relational-model proof)

01List
Relationships table on the firm's Supply Chain tab

Two tables — Outgoing (this firm → others) and Incoming (others → this firm). Each row: target firm (BiText), relationship type chip, description, source pill, last updated. Empty state: "No relationships recorded — add the first supplier or customer."

02Add
Add-relationship drawer (right side)

Typeahead "Target firm" → searches useFirms({ exclude: currentId }); if no match, banner "Not in system — onboard first?" with a link that opens Flow 1 in a side context. Relationship type from ref_relationship_type. Provenance panel persistent. Validation no self-loop; no duplicate edge with same type.

03Trace
Edge with provenance

Saved relationship row shows the source pill alongside the type chip — the same Provenance treatment as every other record. A small footnote at the bottom of the tab: "These relationships will power the value-chain network view in Phase 2." This is the one place we deliberately reference the deferred Phase 2 dashboard — because the planner is doing the work that backs it, and we want to make that work feel meaningful.

Flow 8 — Edit & version

01Edit
Pre-populated form + required change reason

Form opens with current values. On save, a modal asks for change reason (mandatory, free text capped at 240 chars). For time-series tables (HR / Financial / ESG), the modal explains "This creates a new snapshot — prior period is preserved" with a side-by-side preview of both rows.

02Review
Audit tab — diff view

The /audit route per firm shows a timeline of every change with actor, source, reason, and a DiffView that highlights field-level deltas (red = removed value, green = new). Filter by domain and date. The diff view is the same component a Phase-1.5 reviewer screen will use unchanged.

§10

Mock-data plan & fixtures

Generated with faker, seeded with a curated TH-satellite-adjacent name list, persisted to localStorage.

Volume & coverage

  • ~40 firms total. Bell-curve completeness — 6 are fully populated across all 7 domains (the demo's hero cohort); 12 are partially filled (2–4 domains); the remaining ~22 are bare identity rows (so the planner has something to enrich during a live demo).
  • ~120 supply-chain edges connecting the fully-populated cohort plus selected partial firms — enough density that the deferred Phase-2 network view will visibly have a graph to render.
  • ~6 source records seeded: BOI, DBD, NRIIS, GISTDA, NSTDA research report 2024, Manual entry.
  • ~8 import jobs in history (some succeeded, some with errors) so /import is not an empty screen on first visit.
  • ~30 audit entries across the populated firms.

Realism rules

  • Firm names are plausibly fictional with a Thai/English pair: e.g., บริษัท สตาร์ฟิลด์ แอโรสเปซ จำกัด / Starfield Aerospace Co., Ltd. — never reuse a real company's exact name.
  • Juristic IDs are valid 13-digit format (passes the regex) but use a fake-data prefix block that won't collide with real DBD numbers — flagged in the seed as __fixture: true.
  • TRL values follow a realistic distribution — mostly 4–7, a few at 8–9, a handful nullable to demo the missing-data UI.
  • ESG confidence flags spread across reported / estimated / derived — about 40/35/25 — so the confidence pill is visibly used.
  • Bilingual content is real bilingual for the hero cohort (both TH and EN authored), and TH-only or EN-only for the rest, so the "missing translation" empty-state has real cases to render.

Persistence

On first load, seed JSON hydrates into the in-memory store. CRUD operations write a JSON diff overlay to localStorage. Closing the tab and returning lands on the same state, including in-flight import jobs and form drafts. A Reset demo data action in the header dev menu drops the overlay; useful before each live demo. satdb:overlay, satdb:import:<jobId>, satdb:firmdraft, satdb:prefs are the four localStorage keys used.

Defaults for the deferred content questions

Per your direction: skip the open questions, use sane defaults, ensure the UI doesn't commit to any wrong answer.
  1. Value-chain sub-segmentation taxonomy. Phase 1 ships an editable seed taxonomy modeled on SIA's segmentation (Upstream: payload manufacturing, bus manufacturing, AIT services, propulsion, structural components; Midstream: ground systems, launch services, integration; Downstream: data services, applications, value-added services). Loaded from lib/data/seed/ref_valuechain.json — swap a single file to adopt a different taxonomy. UI marks taxonomy items with a subtle PROPOSED chip in the taxonomy admin view so stakeholders see this is a default, not a decision.
  2. Juristic-ID natural key. Defaulted to Thailand's 13-digit DBD juristic person number. The juristic_id field uses regex ^\d{13}$. If the project later adopts a different natural key, the field stays — only the validator changes.
  3. Approval depth. Single-role demo per your call. The schema already carries status ∈ {draft, pending_review, published, retired} so when Phase 1.5 adds reviewers, no migration is needed.
  4. Confidentiality tiers. Defaulted to row-level for Phase 1 — every record carries a confidentiality ∈ {public, restricted, confidential} field, rendered as a small chip. No per-field access_level column yet; if Phase 2 needs field-level masking, the migration is "add a column," not "rebuild the schema."
  5. Priority source agencies for bulk import. Phase 1 ships templates for the four most-cited: BOI, DBD, NRIIS, GISTDA. Each template uses the project's canonical column names; the demo shows column-name mismatches landing as UNKNOWN_COLUMN errors rather than silent failures.
Discipline Every defaulted choice has a single-source-of-truth file the steering committee can amend without code changes: lib/data/seed/ref_valuechain.json, lib/data/seed/ref_relationship_type.json, lib/data/seed/ref_esg_cert.json, etc. We are choosing today's defaults but reserving Phase 2's right to change them.
